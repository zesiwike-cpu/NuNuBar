# 已验证的成功路径

NuNuBar 采用“成功路径优先”的支持口径。只有准确的硬件与电脑组合完成安装、
状态灯、断线恢复和正常键盘功能检查后，才能称为**已验证**。仅仅编译成功或
USB PID 相同，不代表用户可以成功使用。

## 三条已验证路线

| 电脑与键盘 | 状态 | 推荐路线 |
| --- | --- | --- |
| Apple Silicon Mac、macOS 14+、Air65 V3 `19F5:102B` | 有线 Codex 状态灯与可选旋钮映射已实机验证 | 使用官方 USB 控制接口；不刷固件、不进入 DFU |
| Apple Silicon Mac、macOS 14+、Air75 V3 `19F5:1028` | 官方有线侧灯控制已实机验证 | 官方 `1.0.14.6` 或更高；仅旧版本经单独确认走 NuPhyIO 官方升级 |
| Apple Silicon Mac、Air96 V2 ANSI `19F5:3266` | 自定义固件 v7 已验证 | 先测试现有固件；只有失败才准备备份、恢复镜像和两次刷写确认 |

Air60 V2、Air75 V2、Halo75 V2、Windows、Intel Mac、其他 NuPhy 系列和表外
配列都不是普通用户成功路线。仓库可以保留贡献者资产，但 Codex 不得在日常配置
中选择它们。

## Codex 如何按实际情况选择路线

1. 记录操作系统、架构、键盘上印刷的准确型号和实体配列。
2. 检测应用模式 USB VID/PID、HID usage 和报告长度。
3. 必须同时准确匹配本页支持矩阵和 `AGENTS.md` 白名单中的一项。
4. `setupPlan.path` 必须是 `air65-v3-macos-wired`、
   `air75-v3-macos-wired-1.0.14.6` 或 `air96-v2-ansi-macos-v7`，否则停止。
5. App 安装、Hooks 写入、进入 DFU 和固件写入分别获得确认。
6. 最后逐项完成成功清单。缺少任何必需观察时，不得把配置汇报为成功。

## Air65 V3 已验证路线

这条路线已在 macOS arm64、Air65 V3 `19F5:102B`、HID usage
`0001:0000` 和 64 字节输入/输出报告上完成实机验证。当前成功结果包括 App
自动读取活动灯光配置、真实 Codex 状态切换、USB 拔插恢复，以及可选的实体
`F23` 右旋切换下一个 Codex 任务。准确前置条件和证据记录在
[`AIR65_V3_VERIFICATION.md`](AIR65_V3_VERIFICATION.md)。

1. 将 Air65 V3 切到有线 USB 模式，并使用支持数据传输的线材。
   蓝牙可以用于打字，但在当前版本中不能作为 Air65 V3 状态灯通道。
   实机测试确认：标准蓝牙 LED 报告可以成功写入，但左右侧灯都不会变化。
2. 安装当前 NuNuBar macOS 安装包。优先使用 Apple 已公证 DMG；
   文件名包含 `UNNOTARIZED` 的资产属于开发包，必须走明确的 Gatekeeper 确认流程。
3. 启动 NuNuBar 并允许“输入监控”。NuNuBar 只写 HID 输出，不读取或保存按键。
4. 在“键盘”页面确认显示 `Air65 V3`、USB 和已读取的活动灯光配置。该路线
   不会要求 VIA 备份、恢复固件、DFU 或键盘固件写入。
5. 在“Agent”页面接入 Codex，检查局部 Hooks，并在 Codex 设置中手动批准。
6. 先在“键盘”页面运行三色灯光自检，再用真实 Codex 任务触发工作中、需要
   确认和完成。自检命令发送成功不等于肉眼验收，用户仍需确认实体灯光。
7. 运行内置助手的只读 `describe`，然后再次触发状态；灯光必须仍然变化。
8. 拔插一次 USB。NuNuBar 必须在不重启 App 的情况下重新发现同一接口并恢复
   当前状态。

可选的 Air65 V3 黄色键 Fn/地球键转换已使用官方 Karabiner-Elements 16.1.0
单独完成实机验证。它不是状态灯的必需条件，也不会改变任何 V2 固件状态。详见
[`AIR65_V3_FN_SHORTCUT.zh-CN.md`](AIR65_V3_FN_SHORTCUT.zh-CN.md)。

## Air75 V3 已验证路线

Air75 V3 使用官方 `19F5:1028`、HID `0001:0000`、64 字节有线控制接口。
NuNuBar 会读取官方固件版本，并要求不低于 `1.0.14.6`。实机上的 `1.0.13.6`
虽然会回复已收到侧灯写入，却不会应用；NuPhyIO 也复现了同样结果。通过 NuPhyIO
备份、官方升级、重启和配置恢复后，官网与 NuNuBar 均可正常改变侧灯。

配置前阅读 [`AIR75_V3_VERIFICATION.zh-CN.md`](AIR75_V3_VERIFICATION.zh-CN.md)。
该路线绝不使用 Air75 V2 镜像。NuNuBar 运行时应关闭 NuPhyIO，因为两者使用同一
设备级 HID 会话。

NuNuBar 0.15.0 还提供型号独立的 75% 键位映射编辑器，详见
[`AIR75_V3_KEY_MAPPING.zh-CN.md`](AIR75_V3_KEY_MAPPING.zh-CN.md)。准确
`19F5:1028` Karabiner 规则生成已经通过自动化测试；只有实机观察到
`F21`/`F22`/`F23` 和至少一个真实映射动作后，才能把 Air75 旋钮路线标为已验证。

## Air96 V2 ANSI 已验证路线

Air96 V2 使用另一条路径。App 必须先测试现有固件；橙、绿、红三种状态可见时，
说明已经兼容，禁止重复刷写。只有自检失败，才进入型号锁定的 v7 分支；该分支
要求 VIA 备份、准确官方恢复固件、哈希校验、用户手动进入 DFU，以及紧挨写入前
的第二次独立确认。

准确固件文件、SHA-256、所需材料和验收清单见
[`AIR96_V2_SUCCESS.zh-CN.md`](AIR96_V2_SUCCESS.zh-CN.md)。

## 成功验收清单

已验证配置必须完成所有适用项目：

- App 显示准确支持型号和预期连接方式；
- 正常输入、修饰键、Caps Lock、休眠和唤醒不受影响；
- 待机、工作中、需要确认/错误、完成在实体灯光上可区分；
- 自定义颜色和灯效到达全部目标灯区；
- 真实 Codex 生命周期事件可以改变灯光，而不只是演示命令有效；
- Air65 V3 执行只读检测后，后续状态仍可正常发送；
- Air75 V3 在最终验收前显示官方固件 `1.0.14.6` 或更高；
- USB 拔插后无需重启 NuNuBar 即可恢复当前状态；
- 其他 Hooks 和用户配置保持不变；
- 用户知道卸载方式；V2 固件用户还必须掌握官方恢复路线。

缺少的实体检查应明确记录为待验证。不能因为另一个尺寸的键盘成功，或拿到
其他用户的二进制文件，就把测试型号升级为已验证。

## 证据与排查

Air65 V3 的协议、压力、会话和重连证据记录在
[`AIR65_V3_VERIFICATION.md`](AIR65_V3_VERIFICATION.md)。固件证据按型号保存在
`firmware/<model>/BUILD_RECORD.md`。

Air65 V3 长时间停在同一颜色时：

1. 确认运行的是 `/Applications/NuNuBar.app` 中的正式安装版本；
2. 确认设备仍为 `19F5:102B`、usage `0001:0000`，且页面已读出活动配置；
3. 运行“灯光自检”。若自检也不变，问题在 HID 会话/活动配置，不在 Hooks；
4. 若自检正常但 Codex 不变，检查 Hooks 是否指向已安装助手且已批准；
5. 拔插一次 USB，等待页面恢复就绪，再重复自检和真实 Codex 状态。

不要通过进入 DFU 或刷写 V2 固件来解决 Air65 V3 连接问题。

Air75 V3 应先在 NuNuBar 中检查固件。低于 `1.0.14.6` 时先在 NuPhyIO 备份，
并只使用经单独确认的官方升级；达到最低版本后，排查 NuNuBar 或 Hooks 前先关闭
NuPhyIO。
