# NuNuBar Codex Micro 模式

本文记录 NuNuBar 参考 Codex Micro 的产品能力、可复用边界和实现路线。目标不是
伪装成 Work Louder 硬件，而是使用 NuPhy 键盘、公开的 Codex 交互入口和 NuNuBar
现有状态通道，提供稳定且可维护的实体控制体验。

## Codex Micro 实际提供什么

OpenAI 官方页面公开了四类核心能力：

1. 六颗 Agent 键分别显示任务的实时状态，并可切换对应任务；
2. 命令键执行接受、拒绝、按住说话、新建任务等 Codex 操作；
3. 摇杆可触发 PR 审查、调试和重构等工作流；
4. 旋钮可调整推理等级。

硬件本身包含 13 个机械按键、触摸传感器、旋转编码器和平面摇杆，支持 USB-C、
蓝牙、macOS 和 Windows。参考：
[OpenAI Supply Co. x Work Louder](https://openai.com/zh-Hans-CN/supply/co-lab/work-louder/)。

本机 Codex 桌面端还表明，Codex Micro 使用专用 Work Louder HID 服务直接接收
按键、摇杆和旋钮事件，并把内部 Codex 命令与最多六个任务的状态直接发送给设备。
这是 Codex Micro 比普通系统快捷键更可靠的关键原因。该实现不是 NuNuBar 可以
依赖的公开第三方 API。

## NuNuBar 可以复刻的能力

| 能力 | Air65 V3 路线 | 结论 |
| --- | --- | --- |
| 当前 Codex 状态灯 | 现有官方 USB 侧灯控制 | 已验证 |
| App 内设置颜色和灯效 | 现有 NuNuBar 灯光页面 | 已实现 |
| App 内设置实体键动作 | F13-F24 载体键 + Karabiner + Codex 快捷键 | 可实现 |
| 新建任务、打开设置、启动听写 | 使用 Codex 已公开或可配置的快捷键 | 可实现 |
| 增减推理等级、切换计划模式 | 用户先在 Codex 中绑定专用快捷键 | 可实现 |
| 一键触发固定 Skill 或提示词 | 打开新任务并插入预设内容 | 可实现，需验证稳定入口 |
| 六个任务分别显示状态 | Air65 V3 目前只有统一侧灯控制 | 不能原样复刻 |
| 后台直接接受或拒绝确认 | 外部应用没有公开的上下文安全命令 API | 暂不提供 |
| 旋钮和摇杆交互 | Air65 V3 没有对应物理控件 | 使用按键或组合键替代 |

## 为什么不能直接复制原版

Codex Micro 的按键不是简单地发送 Enter、Escape 或 Command-N。Codex 桌面端会先
判断当前命令是否可用，再执行内部动作。例如“接受”只会在确认界面存在时执行。
NuNuBar 如果把普通按键粗暴转换成 Enter，可能在没有确认界面时误发送消息。因此，
任何涉及批准、拒绝或修改权限的动作都必须等 OpenAI 提供公开接口，或者使用经过
明确上下文校验的实现。

NuNuBar 不应：

- 冒充 Codex Micro 的 USB VID/PID；
- 复制或依赖 Codex App 内的 Work Louder 私有包；
- 修改 Codex App 包、注入脚本或依赖其内部 DOM；
- 为了按键映射给 Air65 V3 刷写固件。

这些做法会带来许可、签名、安全和版本兼容问题，不适合作为公开 GitHub 项目。

## 推荐架构

### 1. 状态通道

继续使用现有 Codex Hooks。`AgentState` 已按 `provider + sessionID` 保存多任务记录，
当前只把它们按错误、等待、工作、完成、空闲的优先级合成为一个侧灯状态。

第一版保留这个聚合灯效。后续可增加任务列表和“选中任务”概念，但在 Air65 V3
没有独立可寻址状态灯之前，不宣称六任务同时显示。

### 2. 输入通道

NuPhyIO 把选定物理键保存为不常用的 `F13-F24` 载体键。NuNuBar 负责生成严格限定
到准确键盘 VID/PID 的 Karabiner 规则，并在 App 中显示每个载体键对应的动作。

第一阶段沿用已经实机验证的黄色 `PGDN -> F24` 路线。扩展到多个按键时，用户只需
在 NuPhyIO 中做一次载体键分配，之后的动作选择都在 NuNuBar 中完成。

### 3. 动作通道

每个动作必须标记可靠等级：

- **原生快捷键**：Codex 已提供稳定快捷键，例如新建任务和启动听写；
- **用户快捷键**：用户在 Codex“键盘快捷键”中先分配，NuNuBar 再调用；
- **系统动作**：打开 Codex、打开 URL 或运行用户明确配置的快捷指令；
- **不可用**：缺少安全上下文或公开接口时不提供映射。

NuNuBar 不直接修改 Codex 的私有存储。需要 Codex 快捷键时，App 应打开对应设置、
显示待绑定组合，并让用户在 Codex 中确认。

## Air65 V3 第一版验收范围

第一版 Micro 模式只支持有线 Air65 V3 `19F5:102B`，不刷固件：

1. NuNuBar 显示黄色键当前使用的载体键；
2. 用户可在 App 中选择 Fn/地球键、新建任务、启动听写、打开快捷键设置或自定义
   Codex 快捷键；
3. App 备份并局部合并 Karabiner 配置，不覆盖其他规则；
4. Codex 不在前台时，可能产生歧义的动作不执行；
5. 状态侧灯继续按现有颜色、灯效和显示时间工作；
6. 每个动作必须通过实体按键按下、松开和重复使用验收。

## 后续型号

- Air60/75/96 V2 和 Halo75 V2 保留现有灯光与固件路线；
- Micro 模式的动作模型和 Codex 快捷键调度可以共享；
- 每个型号必须单独定义可用载体键和物理位置，不能复制 Air65 V3 的键位；
- 新增映射不改变现有固件验证状态，也不自动触发 DFU 或刷写。

## 达到完整体验还缺什么

要安全实现与 Codex Micro 完全相同的接受、拒绝、任务切换、按住说话和推理控制，
最理想的条件是 OpenAI 提供公开的桌面端外设 API、URL Action 或可验证的命令 IPC。
在此之前，NuNuBar 应优先完成稳定的快捷键子集，并把不可靠动作明确标记为不可用，
而不是依赖容易随 Codex 更新失效的内部实现。
