# 历史 Air60 蓝牙固件指南

> [!IMPORTANT]
> 这是早期 Air60 V2 蓝牙路径的历史指南。当前 NuNuBar 的配置流程、USB Raw HID
> 固件和多型号安全规则以 `AGENTS.md` 与 `docs/CODEX_SETUP.zh-CN.md` 为准。
> 不要用这份历史指南刷写当前 Air60 V2、Air75 V2、Air96 V2 或 Halo75 V2 的 USB v3 固件。

# 让 Codex 或 Claude Code 帮你写、移植和刷写旧版 NuphyBar 固件

本指南适用于能够在本机读写代码和运行终端命令的编码 Agent。普通网页聊天无法直接检测 DFU 设备或刷写键盘。

## 安全边界

把任务分成两个阶段：

1. **可自动执行**：检查型号资料、读取源码、写灯效、运行测试、编译、校验哈希、列出 DFU 设备。
2. **必须人工确认**：确认键盘准确型号、备份 VIA、准备恢复文件、让键盘进入 DFU、授权最终刷写命令。

AI 不应该：

- 根据“看起来像 Air V2”猜型号；
- 把 Air60 V2 固件刷到 Air75 V2/Air96 V2；
- 在没有恢复固件时开始；
- 跳过构建测试或输入文件哈希；
- 修改无线轮询频率来追求动画帧率；
- 通过蓝牙连续发送动画帧；
- 检测到多个 DFU 设备时自动选择一个；
- 在用户没有明确确认 DFU 和型号前运行 `dfu-util -D`。

## 模板一：让 AI 构建并刷入已支持的 Air60 V2

把下面内容复制给 Codex 或 Claude Code：

```text
你正在 NuphyBar 仓库中工作。目标键盘是 NuPhy Air60 V2 ANSI。

请严格使用 firmware/air60-v2 中已有的 stable-v7 源码和构建器，不要重写无线协议，不要使用任何旧实验固件。

阶段 A（现在可以执行）：
1. 阅读 README.zh-CN.md、firmware/air60-v2/README.md、build.sh 和 verify_candidate.py。
2. 确认我提供的 NuPhy 官方 v2.1.5 固件 SHA-256 是 cd0425f548a01416d1c3c25208ff74867fffd20165520c7c2eaa56000ff347bf。
3. 运行 firmware/air60-v2/test.sh。
4. 使用 build.sh 生成 NuphyBar-Air60-V2-stable-v7.bin。
5. 确认最终 SHA-256 是 c573c7939a53994b50f29313744f27f9af30b90cd064f13fc019f87710b89ac0。
6. 报告测试、输入哈希、输出哈希和将要刷写的绝对文件路径。
7. 到此停止，不要刷写。明确要求我确认“Air60 V2 ANSI 已进入 DFU，确认刷入 stable-v7”。

阶段 B（只有收到上述明确确认后）：
1. 运行 dfu-util -l。
2. 只在检测到一个预期的 STM32 DFU 设备时继续；否则停止。
3. 再次打印固件 SHA-256。
4. 执行 dfu-util -a 0 -s 0x08000000:leave -D <已验证固件>。
5. 检查退出码和下载/校验结果，不要把“命令运行过”当作成功。
6. 刷写后让我重新连接蓝牙，依次验证打字、Caps Lock、空闲、工作、等待和完成。

任何型号、哈希或 DFU 设备不匹配都必须停止，不能尝试绕过保护。
```

## 模板二：让 AI 移植到 Air75 V2 或 Air96 V2

```text
你正在为 NuphyBar 适配 NuPhy <准确型号和布局>。这是一个新型号移植，不允许复用 Air60 V2 二进制或固定函数地址。

目标：复用标准 HID LED 状态编码：
- 0x00 空闲，恢复原厂灯效
- 0x01 工作中
- 0x04 等待/错误
- 0x05 完成
- 0x02 始终保留 Caps Lock

工程要求：
1. 先从 NuPhy 官方固件页确认该型号属于 QMK，而不是 NuPhy IO。
2. 获取该型号官方 QMK 源码和官方恢复固件，记录精确版本与 SHA-256。
3. 找到无线模块传入的完整主机 LED 字段、link_mode、右侧灯 LED 索引和官方侧灯刷新顺序，用源码证据记录，不猜测。
4. 优先做源码级 QMK 移植；只有官方稳定版与公开源码明显不一致时，才提出经过签名保护的最小二进制补丁方案。
5. 不加快 RF/UART 轮询，不发送多帧协议；Mac 只发送持久的两位状态，动画全部在键盘本地渲染。
6. 空闲和 USB 模式必须恢复原厂行为；左侧 Caps 指示不得改变。
7. 为状态解码、灯效连续性、LED 边界和超时行为添加主机测试。
8. 编译后报告 Flash/RAM 增量、最终 SHA-256、修改文件和恢复步骤。
9. 在任何实机刷写前停止，要求我明确确认准确型号、布局、官方恢复文件和 DFU 状态。

不要把“同属 Air V2”理解为可以共用 .bin。完成标准是生成该型号专用固件并在实机逐级验证。
```

## 模板三：为 Halo V2 QMK 或 Gem80 设计灯效

```text
请为 NuphyBar 适配 NuPhy <准确型号>，但不要模仿 Air60 V2 五格侧灯。

先审查该型号实际可控区域（Halolight、铭牌、侧灯或底灯）和 LED 拓扑，然后提出三种能与原厂待机效果明显区分、但不过度闪烁的方案：工作中、等待/错误、完成。状态编码仍使用 0x00/0x01/0x04/0x05，动画必须在固件本地运行。

先只提交灯效模型与主机测试，不修改无线轮询，不刷写。等我选择方案后再接入该型号的官方刷新链路。
```

## AI 执行刷写前的检查清单

AI 在请求最终确认时应该展示：

```text
键盘型号：NuPhy Air60 V2 ANSI
固件基线：NuPhy official v2.1.5
输入 SHA-256：cd0425...
输出文件：/绝对路径/NuphyBar-Air60-V2-stable-v7.bin
输出 SHA-256：c573c7...
测试：全部通过
恢复文件：已准备
DFU：等待用户手动进入
下一条命令：dfu-util ... -D <绝对路径>
```

任一项为空都不能继续。

## 刷写后的分级验证

1. 不启动 NuphyBar，蓝牙连续打字，确认原厂空闲效果和连接稳定。
2. 只发送一次工作状态，继续打字至少几分钟。
3. 测试等待状态，确认五格琥珀双脉冲。
4. 测试完成状态，确认绿色呼吸。
5. 发送空闲，确认恢复原厂彩虹/电量。
6. 切换 Caps Lock，确认左侧青色提示不受影响。
7. 断开重连，确认键盘与 NuphyBar 自动恢复。

如果出现卡键、灯条冻结、断连或无法休眠，立即停止测试并刷回官方恢复固件。不要用提高超时、吞掉错误或继续加快轮询来“修复”。
